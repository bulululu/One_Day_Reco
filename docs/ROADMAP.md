# OneDayReco 开发路线图

## 目标版本：MVP 可直接使用版

用户能完成以下闭环：

1. 注册或本地进入。
2. 选择 MBTI，或通过轻问答得到临时偏好。
3. 输入/获取当前位置天气。
4. 获得一张具体、可执行、符合当下状态的推荐卡。
5. 通过聊天调整方向。
6. 点喜欢、完成、跳过后，下一次推荐能学习。

## 推荐验收标准

每条推荐都必须能回答“我现在下一步做什么”。

- 电影：给出电影名、影院或猫眼检索路径、建议场次、片长、评价。
- 吃饭：给出餐厅/菜系/商圈、人均、口碑、排队或避峰建议。
- 户外：给出具体地点、入口或路线、天气适宜度、预计耗时。
- 居家：给出具体内容、平台/搜索词、预计时长、启动方式。
- 游戏：给出游戏名、平台、单人/多人、预计时长。
- 旅游：给出目的地、交通方式、路程、停留时间、预算。

任何“去看电影”“去吃饭”“出去走走”“打会儿游戏”这类泛泛建议都不算合格。

## Phase 1：真实可用核心链路

目标：一个人打开 App 就能获得可执行推荐。

前端：

- 登录/注册页。
- onboarding 完成后同步用户画像到后端。
- “不知道 MBTI”轻问答结果保存为临时画像。
- 首页当前情况编辑：地点、天气、预算、通勤。
- 推荐卡组件、聊天组件、反馈组件拆分。

后端：

- 用户表、反馈表、推荐历史表。
- 注册/登录/me 接口稳定。
- 推荐请求自动读取用户反馈摘要。
- 反馈写入数据库，并生成用户偏好摘要。
- 天气接口进入推荐过滤。

验证：

- 新用户注册后能完成 onboarding。
- 修改地点天气后推荐内容变化。
- 跳过某类活动后再次推荐降低该类权重。

## Phase 2：活动库扩展

目标：推荐更具体，不像泛泛 AI 回答。

- 扩展活动 schema：城市、天气适配、适合时段、启动门槛、深链模板。
- 活动库从 45 条扩到 150 条。
- 游戏类活动需覆盖休闲、策略、合作、体感、移动端，并包含游戏名称、平台、类型、单人/多人、预计时长。
- 正式上线前，活动和游戏主数据源必须从外部 API 获取；本地 JSON 只保留为无 key 兜底。
- 区分居家、附近、城市探索、短途旅行、屏幕替代活动。
- 每条活动支持 `action_url` 生成规则。
- 增加活动质量检查脚本，避免空字段和泛化推荐。

## Phase 3：屏幕使用时间触发 + 推送服务

目标：从“打开 App 推荐”升级到“合适时机提醒”。

- iOS 接 FamilyControls / DeviceActivity。
- Android 接 UsageStatsManager。
- 触发后调用 `/api/trigger`。
- Expo push token 注册。
- 用户提醒偏好设置。
- 每日推荐提醒。
- 计划开始提醒。
- 测试推送接口。

需要用户确认：

- 是否使用 Expo Push Notifications。
- 是否有 Apple/Android 真机测试环境。

## Phase 4：反馈学习

目标：App 越用越懂用户。

- 反馈明细表。
- 推荐历史表。
- 用户偏好摘要定期更新。
- 规则层排序：跳过降权、喜欢升权、完成强化。
- 对话中的“太远了”“不想出门”“贵了”自动提取为反馈。

## Phase 5：A/B 测试

目标：验证 UI、推荐文案、排序策略。

- 实验分组表。
- 推荐策略版本号。
- 曝光、点击、喜欢、完成事件埋点。
- 简单实验报表接口。

MVP 先不做复杂平台，只记录足够的数据。

## 近期执行顺序

1. 拆分 ChatScreen 剩余模块。
2. 继续压低推荐卡的信息噪音，让视觉保持生活方式首页而不是点评平台。
3. 接入推送 token 注册和测试推送。
4. 增加推荐历史表和用户点击/完成事件。
